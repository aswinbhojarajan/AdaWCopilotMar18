Use this as a master prompt in Replit Agent for Ada’s data setup.

```text
You are working on the Ada Wealth Copilot prototype. Your task is to upgrade the app’s entire demo data foundation so it feels comprehensive, historically rich, realistic, internally consistent, and personalized.

Goal:
Transform the current mock/demo setup into a production-style synthetic dataset and content layer that makes the prototype feel like a real wealth platform used by a real client over time.

Core principles:
1. Everything must be synthetic, privacy-safe, and fictional, but highly believable.
2. Data must be interconnected across database tables, files, APIs, content modules, FAQs, and UI states.
3. The data should not look randomly generated. It should reflect coherent human behavior, market conditions, life goals, portfolio decisions, and engagement history.
4. Personalization must be visible across Home, Wealth, Discover, Collective, and Chat experiences.
5. Historical depth must be sufficient for trends, insights, alerts, comparisons, and narratives.
6. Avoid shallow “demo-only” filler. Every major object should have context, timestamps, relationships, and reason codes where relevant.

Build for:
- A premium wealth client persona with realistic background, financial profile, goals, account relationships, holdings, cash movements, and engagement patterns
- A relationship-manager / advisory context
- A modern wealth app with portfolio intelligence, discovery/news, collective insights, and conversational assistant features

What to create or improve:
A. Database / Seed Data
B. Supporting JSON / mock API files
C. FAQ / help / glossary content
D. Insight-generation inputs
E. Alerts / notifications / nudges
F. Conversation memory / sample prompts / assistant context
G. Empty, loading, warning, and edge-case states
H. Sample documents / reports / summaries where useful

-----------------------------------
1. CLIENT PERSONA AND PERSONALIZATION
-----------------------------------

Create 1 primary flagship synthetic user persona and 2–3 secondary supporting personas for comparison / household / segmentation logic where useful.

For the primary user, define:
- Name, age band, family status, location, residency/tax context
- Occupation / business context
- Risk profile
- Liquidity preference
- Investment sophistication
- Goal priorities
- Key life events
- Preferred sectors / countries / themes
- Banking relationships
- Advisory style preference
- Behavioral tendencies (e.g. cautious in drawdowns, likes dividend income, prefers USD exposure, interested in GCC/India/US markets, etc.)

The primary user should feel like a real affluent Middle East-based client with cross-border exposure and a non-trivial financial life.

Personalization should cascade into:
- Portfolio composition
- Account mix
- Goals
- News relevance
- Alerts shown
- Suggested actions
- FAQs surfaced
- Chat prompts and assistant responses
- Discover feed content
- Collective/community comparisons
- Advisor notes and recommendations

-----------------------------------
2. HISTORICAL DATA DEPTH
-----------------------------------

Create enough history for trend views, insights, seasonality, and narrative analysis.

Minimum historical expectations:
- Portfolio valuation history: at least 36 months monthly, and 12 months daily
- Transactions: at least 18–24 months with realistic cadence
- Security price history used by the app: enough to support 1D / 1W / 1M / 6M / 1Y / 3Y views
- Cash balances and flows: at least 12 months
- Goals progress history: at least 12–24 months
- User engagement / app activity / advisor interactions: at least 6–12 months
- Alerts and notifications history: at least 3–6 months
- Discover/news feed archive: at least several weeks to months, with freshness signals
- Documents/statements/reports: enough to show continuity over time

Historical data must reflect plausible life and market evolution, not flat synthetic curves.

-----------------------------------
3. DATA DOMAINS TO COVER
-----------------------------------

Populate or enhance data for the following domains wherever they exist in the repo. If tables/files do not exist, create appropriate structures and seed files.

A. User / Household
- User profile
- Household members / dependents if supported
- KYC-style summary fields
- Preferences
- Currency preferences
- Risk and suitability summary
- Service tier / advisor assignment

B. Accounts
- Bank accounts
- Brokerage / custody accounts
- Retirement / pension style accounts if relevant
- Deposits / cash management
- External linked accounts via open banking simulation
- Credit / liabilities if relevant to net-worth views

C. Holdings / Portfolio
- Equities
- ETFs / mutual funds
- Bonds / sukuk / fixed income
- Cash / money market
- Alternatives if supported
- Concentration by sector, geography, asset class, currency
- Cost basis, market value, gain/loss, income yield, allocation weights

D. Transactions
- Contributions
- Withdrawals
- Buys / sells
- Dividends / coupons
- Fees
- FX conversions
- Transfers between accounts
- Goal-linked contributions
- Advisor-recommended rebalancing transactions

E. Goals
- Emergency fund
- Education
- Retirement
- Travel / home / milestone goals
- Goal targets, dates, progress, projected probability, funding gaps
- Goal updates over time

F. Insights / Health Check
- Diversification issues
- Cash drag
- Concentration risk
- FX exposure
- Performance attribution
- Goal funding adequacy
- Income stability
- Risk drift
- Rebalancing needs
- Tax/fee awareness where appropriate

G. Discover Feed
- Personalized short-format market/news items
- Events linked to portfolio holdings/themes
- Mixed content types: short brief, explainer, watchlist note, video placeholder, chart card
- Reasons for recommendation (“Because you hold X”, “Relevant to your India exposure”, etc.)

H. Collective
- Community / peer insights using anonymous synthetic cohorts
- Benchmark comparisons
- “Investors like you” patterns
- Trending themes
- Popular watchlist items
- Common mistakes / actions
- Do not overdo gamification; keep it premium and credible

I. Advisor Layer
- Relationship manager profile
- Meeting notes
- Review summaries
- Recommended actions
- Follow-ups
- Documented rationale for advice
- Scheduled check-ins

J. Chat / Assistant Context
- Sample conversation starters
- Saved topics
- Prior assistant memory snippets
- Suggested follow-up questions
- Context cards that explain why the assistant knows something
- FAQs mapped to likely user intents

K. FAQs / Help / Glossary
- Product FAQs
- Portfolio explanation FAQs
- Goal planning FAQs
- Risk FAQs
- Fees / transfers / deposits / withdrawals
- Account linking / open banking
- Security / privacy
- Market terminology
- Why a recommendation was shown
- Why an alert was triggered

-----------------------------------
4. REALISM REQUIREMENTS
-----------------------------------

Make the data feel lived-in.

Use the following realism rules:
- Portfolio should not be perfectly balanced unless there is a reason
- Cashflows should have patterns: salary/business income, periodic top-ups, fees, subscriptions, travel, family expenses, transfers, irregular events
- Holdings should reflect a believable investor journey, not a textbook portfolio
- Some positions should be legacy, some recently added, some underperforming, some winners
- Add realistic market-driven valuation changes over time
- Add occasional dormant periods, bursts of activity, and event-based actions
- Not every recommendation should have been accepted
- Include a few “messy reality” cases: idle cash, overlapping funds, home-country bias, concentrated employer/sector exposure, unexecuted suggestions
- Discover content should connect to actual holdings, geography, sectors, goals, or behavior
- Collective insights should be plausible but anonymized and not overly specific
- FAQs should answer realistic user confusion, not just generic marketing language

-----------------------------------
5. PERSONALIZED STORYLINE
-----------------------------------

Create a coherent backstory for the primary user so the data tells a believable story.

Example storyline dimensions to embed:
- Recently increased savings/investing discipline
- Has cross-border exposures and multiple currencies
- Is balancing long-term wealth building with medium-term goals
- Has some legacy holdings and some newer advisor-led investments
- Is interested in market opportunities but values downside protection
- Has a few unresolved portfolio issues that Ada can help surface
- Uses the app frequently for summaries and explanations, less frequently for deep research
- Has occasional advisor meetings and receives nudges/recommendations

The data should enable the UI to “tell this story” without explicitly dumping it everywhere.

-----------------------------------
6. EDGE CASES AND STATES
-----------------------------------

Also create data and content for non-happy paths:
- Missing linked account / partial sync
- Stale market data warning
- Goal off-track
- Concentrated position alert
- Negative monthly performance
- Deposit pending
- Duplicate watchlist item
- Incomplete profile / risk review due
- No news for a niche holding
- Empty state for a new feature
- Advisor unavailable / next meeting pending
- Portfolio up overall but one goal behind
- Large cash balance waiting for deployment
- FX impact causing confusion
- Suggested action dismissed or snoozed

Ensure the UI has enough seeded cases to demonstrate these states intentionally.

-----------------------------------
7. FILES AND CONTENT TO UPDATE
-----------------------------------

Update the repo comprehensively, not just one database file.

Where relevant, create or update:
- Database seed scripts
- Mock JSON data files
- Static content files
- FAQ markdown / JSON / CMS-like content
- Sample API response files
- Demo documents / reports / snapshots
- Recommendation explanation files
- Alert templates
- Notification content
- Persona metadata
- Cohort/benchmark reference datasets
- Test fixtures used by the UI

Do not leave important demo content half-populated.

-----------------------------------
8. CONSISTENCY RULES
-----------------------------------

Maintain strict consistency across all layers:
- Totals in UI-facing summaries must match underlying data
- Transactions must reconcile with balances and holdings changes
- Allocation percentages must sum correctly
- Performance numbers must be plausible relative to price history
- Alerts must be explainable from the data
- Recommendations must align with portfolio issues
- News relevance tags must map to user profile/holdings/themes
- FAQs must reflect actual product behaviors shown in the app
- Advisor notes should reference real positions, goals, or events in the seeded data

Add validation checks or helper scripts if useful.

-----------------------------------
9. OUTPUT EXPECTATIONS
-----------------------------------

When you make changes, do all of the following:
1. Inspect the current repo structure first
2. Reuse and extend existing schemas/files where possible
3. Create missing structures only where necessary
4. Seed comprehensive synthetic data
5. Update related FAQs/content files
6. Ensure the app renders meaningful states across tabs
7. Add comments/doc notes where helpful for maintainability

Also generate a concise changelog that explains:
- What files were updated
- What new datasets were added
- What assumptions were made
- Which parts are primary-user personalized
- Any known gaps or recommended next enhancements

-----------------------------------
10. QUALITY BAR
-----------------------------------

The final result should make a stakeholder believe:
- this is a serious prototype
- this user has a real financial life
- the app understands the user personally
- the insights come from real underlying data
- the product has depth, history, and credibility

Do not produce toy demo data.
Do not produce flat random placeholders.
Do not leave generic lorem ipsum content.
Do not create internally inconsistent numbers.

Proceed by first auditing the current repo/data setup, then upgrading it end-to-end.
```
